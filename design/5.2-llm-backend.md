# 5.2 可配置 LLM 后端

本文档详细描述 `mcp2cli generate cli` 和 `mcp2cli generate skill` 的 LLM 后端配置。

> 源自 [5.0-cli-generation.md](5.0-cli-generation.md) 的 5.6 节。

---

## MVP 范围

MVP 阶段只实现 `claude-cli` 后端。Python 实现中使用 `LLMBackend` 抽象接口，便于后续接入其他后端。

## 配置文件

`~/.agents/mcp2cli/config.yaml`：

```yaml
llm:
  # 后端类型: claude-cli（MVP 仅支持此后端）
  # 未来计划支持: codex | anthropic-sdk | openai-compatible
  backend: claude-cli

  # claude-cli 配置（默认）
  claude_cli:
    command: claude           # claude 可执行文件路径
    model: opus               # 模型选择: opus | sonnet | haiku
```

## 接口抽象设计

Python 实现中定义 `LLMBackend` 抽象基类，MVP 只实现 `ClaudeCLIBackend`：

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class LLMResult:
    """LLM 调用返回结果"""
    session_id: str | None    # 用于会话续接
    result: str               # LLM 输出文本
    is_error: bool            # 是否出错

class LLMBackend(ABC):
    """LLM 后端抽象接口"""

    @property
    @abstractmethod
    def backend_type(self) -> str:
        """后端类型标识，用于 session 文件中记录（如 "claude-cli"、"anthropic-sdk"）"""
        ...

    @abstractmethod
    def invoke(self, prompt: str) -> LLMResult:
        """首次调用 LLM"""
        ...

    @abstractmethod
    def resume(self, session_id: str, prompt: str) -> LLMResult:
        """在已有会话中续接调用（用于校验失败后重试）"""
        ...


class ClaudeCLIBackend(LLMBackend):
    """claude -p 后端实现（MVP 默认）"""

    def __init__(self, command: str = "claude", model: str = "opus"):
        self.command = command
        self.model = model

    def invoke(self, prompt: str) -> LLMResult:
        # claude -p "<prompt>" --output-format json --model <model> --dangerously-skip-permissions
        ...

    def resume(self, session_id: str, prompt: str) -> LLMResult:
        # claude -p "<prompt>" --output-format json --resume <session_id>
        ...
```

后续新增后端（如 `CodexBackend`、`AnthropicSDKBackend`）只需实现 `LLMBackend` 接口即可，无需修改生成逻辑。

## claude-cli 调用方式

```bash
# 首次调用 — 返回 JSON 含 session_id
claude -p "<assembled_prompt>" \
  --output-format json \
  --model opus \
  --dangerously-skip-permissions

# 校验失败时 — 在同一会话中续接
claude -p "<error_prompt>" \
  --output-format json \
  --dangerously-skip-permissions \
  --resume <session_id>
```

- `--output-format json`：获取结构化响应，含 `session_id` 用于会话续接
- `--dangerously-skip-permissions`：跳过所有权限审批，确保自动化流程不被交互式确认阻塞。LLM 操作范围受限于 prompt 指令（读取模板/tools 文件、写入 YAML、执行 `mcp2cli validate`），由程序侧控制安全边界

## Session 管理

> 参考实现：[claude-code-bridge/src/claude/session.py](https://github.com/user/claude-code-bridge/blob/main/src/claude/session.py)

### Session ID 获取

`claude -p --output-format json` 返回结构化 JSON，其中包含 `session_id`：

```json
{
  "type": "result",
  "session_id": "abc12345-6789-abcd-ef01-234567890abc",
  "result": "Generated: 12 tools mapped to 18 commands",
  "is_error": false
}
```

程序从返回 JSON 中提取 `session_id`，存入 `LLMResult.session_id`。

参考 claude-code-bridge 的 `_handle_session_id` 方法（session.py L577-580）：Claude CLI 在 `system`、`assistant`、`result` 等消息类型中都可能携带 `session_id` 字段。对于 mcp2cli 使用的 `--output-format json` 模式，`session_id` 固定出现在返回的顶层 JSON 中。

### Session 续接

`ClaudeCLIBackend.resume()` 通过 `--resume <session_id>` 在同一会话中续接对话：

```bash
claude -p "<error_prompt>" --output-format json --resume <session_id>
```

续接时 LLM 保留完整上下文（已读取的规则、示例、tools 文件、生成过的 YAML），只需传入错误信息即可针对性修复，无需重新读取文件。Token 消耗约为首次调用的 10-20%。

参考 claude-code-bridge 的 `_start_process` 方法（session.py L194-200）中 `--resume` 的用法。

### Session 持久化存储

`generate cli` / `generate skill` 流程可能涉及多次 LLM 交互（首次调用 → 校验失败 → 重试）。为防止进程崩溃或用户 Ctrl+C 后 session 丢失，将 session_id 持久化到临时文件，支持断点续接。

**存储位置**：`~/.agents/mcp2cli/.sessions/`

**文件名**：`<command>-<server>.session.json`

**文件内容**：

```json
{
  "session_id": "abc12345-6789-abcd-ef01-234567890abc",
  "backend": "claude-cli",
  "command": "generate cli",
  "server": "mcp-atlassian",
  "created_at": "2026-04-03T10:00:00Z",
  "last_used_at": "2026-04-03T10:05:00Z",
  "status": "in_progress"
}
```

**`backend` 字段说明**：记录创建 session 时使用的 LLM 后端类型。resume 时据此选择正确的后端接口进行续接（不同后端的续接方式不同：claude-cli 用 `--resume`，anthropic-sdk 用 messages 数组追加）。如果当前配置的后端与 session 文件中的 `backend` 不一致，提示用户并询问是否放弃旧 session 从头开始。

**生命周期**：

| 时机 | 操作 |
|------|------|
| `invoke()` 成功返回 | 从 LLM 返回的 JSON 中提取 `session_id`，写入 session 文件（`status: "in_progress"`） |
| `resume()` 调用前 | 读取 session 文件获取 `session_id`，更新 `last_used_at` |
| 命令成功完成 | 删除 session 文件 |
| 用户再次执行同一命令 | 检测到已有 session 文件 → 提示是否续接 |

**断点续接场景**：

```bash
# 首次执行，中途 Ctrl+C 中断
$ mcp2cli generate cli mcp-atlassian
Generating... ^C  # 中断，session 文件已写入

# 再次执行，检测到未完成的 session
$ mcp2cli generate cli mcp-atlassian
Found unfinished session from 2026-04-03T10:00:00Z. Resume? [Y/n] y
Resuming session abc12345-...
```

**过期与清理**：

- 超过 24 小时的 session 文件自动忽略（视为过期），不再提示续接
- `--no-resume` 参数可强制从头开始，忽略已有 session 文件

**设计动机**：

参考 claude-code-bridge 的 session 管理思路（session.py L114-137 `_find_latest_session_id`），但 mcp2cli 不依赖 Claude CLI 内部的 `~/.claude/projects/` 目录结构，而是自行管理更轻量的映射文件。原因：

1. `~/.claude/projects/` 的目录结构是 Claude CLI 内部实现细节，可能随版本变化
2. mcp2cli 需要按 "命令 + server" 维度精确匹配 session，而非按目录扫描最新文件
3. 自行管理的 session 文件可携带额外元数据（command、status、时间戳）

### ClaudeCLIBackend 完整实现

```python
import json
import subprocess
from dataclasses import dataclass
from datetime import datetime, timezone
from pathlib import Path
from typing import Optional

SESSIONS_DIR = Path.home() / ".agents" / "mcp2cli" / ".sessions"
SESSION_EXPIRY_HOURS = 24


@dataclass
class SessionFile:
    """Session 持久化文件的数据模型"""
    session_id: str
    backend: str              # LLM 后端类型（如 "claude-cli"、"anthropic-sdk"）
    command: str
    server: str
    created_at: str
    last_used_at: str
    status: str  # "in_progress" | "completed"

    def to_dict(self) -> dict:
        return {
            "session_id": self.session_id,
            "backend": self.backend,
            "command": self.command,
            "server": self.server,
            "created_at": self.created_at,
            "last_used_at": self.last_used_at,
            "status": self.status,
        }

    @classmethod
    def from_dict(cls, data: dict) -> "SessionFile":
        return cls(**data)


class ClaudeCLIBackend(LLMBackend):
    """claude -p 后端实现（MVP 默认）"""

    def __init__(self, command: str = "claude", model: str = "opus"):
        self.command = command
        self.model = model

    @property
    def backend_type(self) -> str:
        return "claude-cli"

    def invoke(self, prompt: str, *, command_name: str = "", server_name: str = "") -> LLMResult:
        """首次调用 LLM"""
        cmd = [
            self.command, "-p", prompt,
            "--output-format", "json",
            "--model", self.model,
            "--dangerously-skip-permissions",
        ]
        result = self._run(cmd)

        # 持久化 session_id
        if result.session_id and command_name and server_name:
            self._save_session(result.session_id, command_name, server_name)

        return result

    def resume(self, session_id: str, prompt: str) -> LLMResult:
        """在已有会话中续接调用"""
        cmd = [
            self.command, "-p", prompt,
            "--output-format", "json",
            "--dangerously-skip-permissions",
            "--resume", session_id,
        ]
        result = self._run(cmd)

        # 更新 session 文件的 last_used_at
        # （session_id 不变，无需重写整个文件）

        return result

    def find_session(self, command_name: str, server_name: str) -> Optional[str]:
        """查找未过期的、未完成的、后端匹配的 session_id"""
        path = self._session_path(command_name, server_name)
        if not path.exists():
            return None

        try:
            data = json.loads(path.read_text())
            sf = SessionFile.from_dict(data)

            # 检查是否过期
            created = datetime.fromisoformat(sf.created_at)
            if (datetime.now(timezone.utc) - created).total_seconds() > SESSION_EXPIRY_HOURS * 3600:
                path.unlink(missing_ok=True)
                return None

            if sf.status != "in_progress":
                return None

            # 检查后端类型是否匹配
            if sf.backend != self.backend_type:
                # 后端不匹配，无法续接（不同后端的 session 续接方式不同）
                return None

            return sf.session_id
        except (json.JSONDecodeError, KeyError, ValueError):
            path.unlink(missing_ok=True)
            return None

    def clear_session(self, command_name: str, server_name: str) -> None:
        """命令成功完成后清理 session 文件"""
        self._session_path(command_name, server_name).unlink(missing_ok=True)

    # -- 内部方法 --

    def _run(self, cmd: list[str]) -> LLMResult:
        """执行 claude CLI 并解析 JSON 返回"""
        proc = subprocess.run(cmd, capture_output=True, text=True)
        try:
            data = json.loads(proc.stdout)
            return LLMResult(
                session_id=data.get("session_id"),
                result=data.get("result", ""),
                is_error=data.get("is_error", False),
            )
        except json.JSONDecodeError:
            return LLMResult(session_id=None, result=proc.stdout, is_error=True)

    def _save_session(self, session_id: str, command_name: str, server_name: str) -> None:
        """将 session_id 写入临时文件"""
        SESSIONS_DIR.mkdir(parents=True, exist_ok=True)
        now = datetime.now(timezone.utc).isoformat()
        sf = SessionFile(
            session_id=session_id,
            backend=self.backend_type,
            command=command_name,
            server=server_name,
            created_at=now,
            last_used_at=now,
            status="in_progress",
        )
        self._session_path(command_name, server_name).write_text(
            json.dumps(sf.to_dict(), indent=2)
        )

    def _session_path(self, command_name: str, server_name: str) -> Path:
        """生成 session 文件路径"""
        # "generate cli" + "mcp-atlassian" → "generate-cli-mcp-atlassian.session.json"
        safe_name = f"{command_name.replace(' ', '-')}-{server_name}"
        return SESSIONS_DIR / f"{safe_name}.session.json"
```

### 调用方 generate 流程集成

```python
# cli_gen.py 中的 generate 主流程（伪代码）
def generate_cli(server: str, backend: LLMBackend):
    command_name = "generate cli"

    # 1. 检查是否有未完成的 session（自动匹配 backend 类型）
    existing_session = backend.find_session(command_name, server)
    if existing_session:
        if prompt_user("Found unfinished session. Resume? [Y/n]"):
            result = backend.resume(existing_session, "<resume_prompt>")
        else:
            existing_session = None

    # 2. 首次调用
    if not existing_session:
        result = backend.invoke(prompt, command_name=command_name, server_name=server)

    # 3. 程序侧校验 + 重试
    errors = validate(server)
    if errors and result.session_id:
        error_prompt = f"校验失败：{errors}\n请修复..."
        result = backend.resume(result.session_id, error_prompt)

    # 4. 成功，清理 session 文件
    backend.clear_session(command_name, server)
```

## 与 claude-code-bridge 的差异总结

| 维度 | claude-code-bridge | mcp2cli |
|------|-------------------|---------|
| 进程模式 | 持久化进程（`--input-format stream-json --output-format stream-json`），适合高频交互 | 单次调用（`claude -p`），适合一次性生成任务 |
| Session ID 获取 | 从 stream 中多种消息类型提取（system/assistant/result） | 从 `--output-format json` 返回的顶层 JSON 提取 |
| Session 发现 | 扫描 `~/.claude/projects/` 下的 `.jsonl` 文件，按修改时间排序 | 自行管理 `.sessions/` 目录，按 "命令+server" 精确匹配 |
| Session 恢复 | 跨会话恢复（适合长期运行的 bot） | 同命令断点续接（适合一次性任务的中断恢复） |
| Session 生命周期 | 长期（直到进程退出或用户切换） | 短期（命令完成即清理，24 小时过期） |

## 未来扩展（非 MVP）

| 后端 | Prompt 模式 | LLM 文件访问 | 权限控制 | 会话续接方式 |
|------|------------|-------------|---------|-------------|
| claude-cli (MVP) | 文件引用模式 | 有（Read/Write/Bash） | `--dangerously-skip-permissions` | `--resume <session_id>` + session 文件 |
| codex | 文件引用模式 | 有（通过工具） | 类似 skip-permissions | `codex resume <session_id>` |
| anthropic-sdk | 内联模式 | 无 | N/A（纯 API 调用） | messages 数组追加（内存中管理） |
| openai-compatible | 内联模式 | 无 | N/A（纯 API 调用） | messages 数组追加（内存中管理） |

内联模式后端需要额外处理：程序预先读取文件内容嵌入 prompt，从响应中提取 YAML，程序侧负责写入文件。这些差异通过 `LLMBackend` 接口的不同实现来封装。内联模式的 session 管理（messages 数组）可序列化到 session 文件中以支持断点续接。
